# Play-Together GBA — serverless edition

Play your Game Boy Advance games **together**, in sync, from a phone or laptop —
with **no server you run or pay for**. The whole app is a static site on GitHub
Pages backed by **Firebase Realtime Database** on the free **Spark** plan. Whoever
holds the controls plays; everyone else watches in sync. Each person loads their
**own** local ROM (hash-checked so everyone runs byte-identical content); ROMs are
never uploaded or shared.

This is the **serverless re-platforming** of the original WebSocket/Docker version
(which still lives on the `server-version` branch). The emulator, sync, speed
control, handoff, and touch controls are unchanged; only the transport, identity,
and ROM handling were swapped onto Firebase. See `docs/SPEC-SERVERLESS.md` for the
full design and `DECISIONS.md` for the choices made.

## What you get

- A public link anyone can open — **no accounts, no login**.
- The owner (whoever created a game) mints **single-use invite links**; each link
  lets exactly one device join, once. Joined devices reconnect forever without a
  new invite (a durable per-device credential, stored in the browser).
- Synced play, speed control (1×–8×), and controller handoff — including when
  someone’s phone drops off cellular mid-game.
- Durable saves that persist across days, stored in your Firebase database.
- Stays inside the Spark free tier for family-scale use (see *Free-tier notes*).

## How it stays free and safe

- Everything lives in **Realtime Database on Spark** — **no Cloud Functions, no
  Cloud Storage for Firebase** (both would force the Blaze plan + a credit card and
  remove the hard quota cap). No surprise bills.
- Security is enforced entirely by **declarative database rules** (`database.rules.json`),
  not server code. The rules are the fence: default-deny, with a three-tier
  capability model (owner / single-use invite / member) and a controller write-lock.

## 1. Firebase setup (one time, ~10 minutes)

1. **Create a project** at <https://console.firebase.google.com> → *Add project*.
   You can disable Google Analytics. Stay on the **Spark (free)** plan — do **not**
   upgrade to Blaze, and do not add a billing account.
2. **Create a Realtime Database**: left nav → *Build → Realtime Database* →
   *Create Database*. Pick a location. Start in **locked mode** (we’ll publish real
   rules in step 4). Note the database URL — it looks like
   `https://YOUR-PROJECT-default-rtdb.firebaseio.com` (or `...REGION.firebasedatabase.app`).
3. **Enable Anonymous Authentication**: *Build → Authentication → Get started →
   Sign-in method →* enable **Anonymous**. (Optional: enable anonymous-account
   auto-cleanup if your console offers it, so stale UIDs don’t accrue.)
4. **Publish the security rules.** Either:
   - paste the contents of `database.rules.json` into *Realtime Database → Rules →
     Publish*, **or**
   - from a clone of this repo: `npm i -g firebase-tools && firebase login &&
     firebase use YOUR-PROJECT-ID && firebase deploy --only database`
     (deploying RTDB rules is free on Spark).
5. **Get your web config**: *Project settings (gear) → General → Your apps →* add a
   **Web app** (`</>`). Copy the `firebaseConfig` object. You need at least
   `apiKey`, `authDomain`, `databaseURL`, `projectId`, `appId`. **This config is not
   a secret** — security comes from the rules, not from hiding it.

Turn that config into the JSON this app reads (note the camelCase keys):

```json
{
  "apiKey": "AIza…",
  "authDomain": "your-project.firebaseapp.com",
  "databaseURL": "https://your-project-default-rtdb.firebaseio.com",
  "projectId": "your-project",
  "appId": "1:…:web:…"
}
```

## 2. Deploy to GitHub Pages

This repo ships a workflow (`.github/workflows/pages.yml`) that builds the static
site and deploys it to Pages on every push.

1. **Enable Pages with the Actions source**: repo *Settings → Pages → Build and
   deployment → Source → GitHub Actions*. (If you cloned/forked, do this once.)
2. **Provide your Firebase config** as a repo **Actions variable** (not a secret —
   it isn’t one, and variables are easier): *Settings → Secrets and variables →
   Actions → Variables → New repository variable*, name **`FIREBASE_CONFIG`**, value
   = the JSON from step 1 above (single line is fine). The workflow writes it into
   the build. *(Alternatively, commit a `client/public/firebase-config.json` — it’s
   git-ignored by default; `git add -f` it if you prefer. It’s not secret.)*
3. **Push** to `main` (or `serverless`). The workflow builds and deploys. Your site
   appears at `https://<you>.github.io/<repo>/` (or your custom Pages domain).
4. **Open it on your phone.** First load registers a service worker and reloads once
   to enable cross-origin isolation (needed by the emulator). Then *Start a new game*.

> The build’s base path is `/<repo>/`. The workflow sets it automatically from the
> repo name. If you serve from a custom domain at the root, set `VITE_BASE=/`.

## 3. Play

- **Start a game** (you become the *owner*): enter your name, pick your local `.gba`
  ROM. Your ROM is fingerprinted (SHA-256) and stored only on your device.
- **Invite people**: open the in-game menu (drag the handle up) → *Create invite
  link* → send it to **one** person. Each link works once.
- **Join**: open the invite link, pick a name, then load **your own copy of the same
  ROM**. If it doesn’t byte-match, you get a clear message — everyone must use the
  exact same ROM version (like a shared mod).
- **Controls/handoff**: the controller has the gamepad; others watch. *Hand over* to
  pass control, or *Take control* when the seat is free. If the controller’s
  connection drops, control passes automatically to the next person.
- **Reconnect**: your device remembers games you’ve joined — just reopen them from
  the home screen. No new invite needed (unless you clear site data / switch device).

## 4. Owner recovery (important footgun)

Ownership is tied to **this device’s** anonymous credential. If you **clear site
data** or switch devices, your next visit gets a *new* anonymous UID and you lose
the ability to mint invites / manage existing games you created.

Because you control the Firebase project, recovery is a one-line console edit
(console writes bypass the rules):

1. In the app (home screen, or the in-game menu → *Your device ID*), copy your
   **current** UID. Keep it somewhere safe in advance if you can.
2. In the Firebase console → *Realtime Database → Data*, navigate to
   `sessions / <sessionId> / meta / owners` and add a child:
   **key = your current UID, value = `true`** (boolean).
3. Reload the app — you’re an owner again.

There is intentionally **no in-app co-owner or token recovery** — the console step
is enough since you hold the project.

## 5. Free-tier notes (Spark)

Verified Spark limits (mid-2026): **~100 simultaneous connections**, **~1 GB
stored**, **~10 GB/month download**, anonymous auth with a **3,000 daily-active-user**
ceiling. Each open tab is one connection. The app keeps you inside these by:

- storing **only the latest** snapshot (overwritten, never appended);
- **pruning** the input/speed relay streams after each snapshot;
- writing a durable checkpoint only every ~30 s;
- letting the owner **End game & delete** a finished session to reclaim storage.

For family-of-family scale (a handful of devices per game, a few games) this sits
comfortably in the free tier. If you routinely approach the caps, see
`docs/SPEC-SERVERLESS.md` §12 for the migration path (the backend lives behind an
adapter so Supabase/Ably could replace Firebase without touching app logic).

## 6. Local development

```bash
npm install
npm run dev            # Vite dev server at http://localhost:5173 (COOP/COEP set for COI)
```

For Firebase locally, run the emulator and point a `client/public/firebase-config.json`
at it (see the example in `client/public/firebase-config.example.json`, add an
`emulators` block):

```bash
npm run itest          # security-rules + transport + sync + guardrails tests, under the Firebase emulator
npm --workspace client test   # unit tests
```

`npm run itest` boots the Firebase emulator, runs the adapter/rules integration
tests, and shuts it down — this is how the rules were adversarially verified.

## Project layout

- `client/` — the static PWA (Vite + React). Emulator core vendored in
  `client/public/emulator/`.
- `client/src/net/` — the backend **adapter seam** (`adapter.ts`) and the Firebase
  RTDB implementation (`firebaseAdapter.ts`), plus the emulator-backed `*.itest.ts`.
- `database.rules.json` — the security rules (the fence).
- `client/public/coi-serviceworker.js` — COOP/COEP shim + app-shell cache (one SW).
- `docs/SPEC-SERVERLESS.md` — the design spec. `PROGRESS.md` / `DECISIONS.md` /
  `QUESTIONS.md` — build log, choices, and open items.

ROMs are never included, hosted, or transmitted. Bring your own.
